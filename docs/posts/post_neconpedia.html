Neconpedia - a knowledge base for NECON

Like many, I was inspired by Andrej Karpathy’s recent post on knowledge databases. I adapted his protocol to make it even simpler and fully no-code (although not as powerful) to build a concept base for my congressional studies research group (NECON). The result is NECONpedia

I document the simplified protocol I followed below:

1. Create a project in Claude with a structured system prompt

Open a new Project in Claude and define project instructions along these lines:

You are my academic knowledge base assistant in political science. Your task 
is to transform papers I upload into a structured wiki in Markdown, ready to 
be opened in Obsidian.

Every time I send papers, you must:

- Read all submitted documents
- Generate the following Markdown files:
  - _INDEX.md — homepage listing all documents and concepts, with [[backlinks]]
  - papers/paper_name.md — a record for each paper including: authors, year,
    abstract (3–5 paragraphs), central arguments, methodology, contributions,
    limitations, and a "Related Concepts" section with [[backlinks]]
  - concepts/concept_name.md — one entry per key concept emerging from the
    collection, including definition, debate in the literature, and an 
    "Open Questions" section
- Use [[backlinks]] in Obsidian format whenever mentioning a concept or paper 
  that has its own page
- At the end, deliver everything as a single .zip file for download, ready to 
  be opened in Obsidian as a vault

Quality standards:

- Write in Portuguese, but keep specific concepts in their original English 
  when appropriate
- Be dense and academic, without oversimplifying
- Cite sources (book/article title and page number for each piece of 
  information)
- Use excerpts in the original language to support different sections
- Prioritize connections between papers — how they relate and where they diverge
- Always include an "Open Questions" section in concept entries

When I send new papers in future sessions, update the index and create new 
backlinks connecting to existing content.

2. Upload your files in batches

Add books and papers in new conversation threads in the project. Because of upload size limits, do this in batches across multiple conversations within the same project. My starting point was the bibliography for our Legislative Studies introductory seminar at IESP.

3. Download the .zip and open it in Obsidian

Claude produces a .zip file containing the full Markdown wiki. Unzip it and open the folder as a vault in Obsidian. You’ll get an index page, individual paper notes, and concept entries — all interlinked with [[backlinks]].

4. Publish the vault (optional)

If you want the wiki to be public, Obsidian Publish makes this straightforward. Our legislative studies base is now available online and updated as we add new readings.

5. Iterate: add new files or make corrections

When new papers arrive, upload the existing .zip back to the Claude project alongside the new PDFs and ask Claude to incorporate them while preserving the structure and updating all backlinks. Then replace the old Obsidian folder with the updated files.

What you get

Each paper becomes a structured note with authors, year, a multi-paragraph summary, central arguments, methodology, contributions, limitations, and a “Related Concepts” section. Each key concept gets its own page with a definition, a review of how it’s debated in the literature, and an “Open Questions” section. The index ties everything together.

Because Claude generates the notes in Portuguese (with English technical terms preserved), the result is directly usable by our research group without any post-editing. The backlinks mean that reading one paper surfaces connections to related concepts and other papers automatically in Obsidian’s graph view.

Limitations and next steps

This approach trades power for simplicity. Unlike Karpathy’s protocol, there’s no local vector store, no programmatic retrieval, and no automated pipeline — each update requires a manual round-trip through Claude. Upload size limits also mean large bibliographies have to be ingested incrementally.

That said, for a research group that wants a shared, readable, interlinked knowledge base without any engineering overhead, it works well. Possible extensions include adding a revision workflow (asking Claude to flag claims that conflict across papers), generating reading guides per theme, or integrating with Zotero exports.
